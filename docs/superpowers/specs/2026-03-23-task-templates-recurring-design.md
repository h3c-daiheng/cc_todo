# 任务模板与周期任务设计

## 概述

在当前待办系统中新增“任务模板 + 周期任务生成”能力，优先解决重复建任务成本高、个人和团队任务创建效率低的问题。第一版支持三类模板：个人模板、团队模板、系统模板；支持个人模板和团队模板配置简单周期规则，由后端定时自动生成任务；系统模板仅支持手动套用，不参与自动生成。

## 设计决策

### D1: 功能边界

第一版聚焦两件事：
- 让用户把常用任务保存为模板，后续快速套用
- 让个人/团队模板可以按简单周期自动生成任务

第一版明确支持：
- 个人模板
- 团队模板
- 系统公共模板
- 每日 / 每周 / 每月三种周期规则
- 从模板手动创建任务
- 后端定时自动生成任务

第一版明确不支持：
- 模板版本历史
- 一个模板多个周期规则
- 导入导出模板
- 复杂日历规则
- 系统模板自动生成任务

### D2: 用户使用流程

#### 个人场景
用户在“我的任务”页面进入模板入口，创建个人模板，填写默认标题、描述、优先级、标签、默认截止偏移天数等信息。之后可以：
- 手动从模板创建任务
- 给该模板配置周期规则，系统按规则自动生成个人任务

#### 团队场景
团队负责人或管理员在“团队任务”页面进入团队模板入口，创建团队模板，指定团队上下文和默认负责人（可选）。之后可以：
- 团队成员查看并手动套用团队模板创建任务
- 团队负责人/管理员为模板配置周期规则，系统自动生成团队任务

#### 系统场景
管理员在后台“系统模板管理”页面维护系统公共模板。普通用户可查看并手动套用系统模板，但不能编辑，也不能为系统模板配置自动周期生成。

### D3: 数据结构和后端设计

建议新增两个核心对象：`task_templates` 与 `template_schedules`。

#### task_templates
建议字段：
- `id`
- `name`：模板名，便于列表识别
- `title`：默认任务标题
- `description`
- `priority`
- `scope`：`personal | team | system`
- `owner_user_id`：个人模板所有者
- `team_id`：团队模板所属团队
- `default_assigned_to`：默认负责人
- `default_due_offset_days`：相对生成时间的默认截止偏移
- `is_active`
- `created_at`
- `updated_at`

标签存储建议沿用现有任务标签的关系型模式，不额外引入 JSON 存储分支。也就是说，模板标签在实现上应尽量复用当前 `TaskLabel` 的思路，避免同一项目里出现两套标签模型。

#### template_schedules
建议字段：
- `id`
- `template_id`
- `enabled`
- `frequency`：`daily | weekly | monthly`
- `hour`
- `weekday`
- `monthday`
- `last_run_at`
- `created_at`
- `updated_at`

#### 自动生成执行方式
复用当前后端 APScheduler，采用“每小时扫描一次”的简单模式：
1. 找出启用中的周期规则
2. 判断当前时间是否命中
3. 通过 `last_run_at` 做基础去重
4. 调用统一的模板实例化逻辑创建任务
5. 更新 `last_run_at`

这里需要补充几个第一版必须明确的执行语义：
- **时区**：第一版统一按服务器本地时区执行，不额外做每用户时区配置
- **扫描窗口**：每小时整点扫描一次，只判断“当前小时是否命中”，不补跑更早错过的小时
- **服务停机恢复**：若服务停机错过了某次生成窗口，第一版不做补生成，恢复后从下一次命中时间继续
- **monthly 非法日期**：若规则设置为 29/30/31，而当月不存在该日期，则该月跳过，不顺延到月末
- **去重语义**：`last_run_at` 表示该 schedule 最近一次成功生成任务的时间戳；同一自然触发窗口内只允许成功生成一次

#### 自动生成任务的归属
由于当前仓库的任务权限强依赖 `created_by`、`assigned_to`、`team_id`，周期生成时必须固定归属规则：
- **个人模板自动生成的任务**：`created_by = owner_user_id`
- **团队模板自动生成的任务**：`created_by = team.created_by`（即生成当下的团队负责人）
- **system 模板**：第一版不允许自动生成，因此不存在对应归属问题

这个设计保证生成后的任务仍然沿用现有权限模型；若团队负责人发生变化，则后续自动生成的新任务自动跟随新负责人，旧任务不回写。

#### 手动套用模板时的归属
手动 `instantiate` 也必须明确 `created_by` 规则，避免和现有任务权限模型冲突：
- **个人模板手动套用**：`created_by = 当前操作用户`
- **团队模板手动套用**：`created_by = 当前操作用户`
- **系统模板手动套用**：`created_by = 当前操作用户`

也就是说，手动套用强调“是谁点了创建，谁就是任务创建人”；自动生成强调“任务归属到模板拥有者或当前团队负责人”。

对于团队模板：
- 若操作用户是普通团队成员，也允许手动套用并成为 `created_by`
- 若同时传入 `assigned_to` 覆盖字段，仍需遵守现有团队任务负责人校验规则，但不影响 `created_by` 归属

这样可以最大限度复用当前任务权限模型，不需要为模板场景发明新的任务归属规则。

#### 默认负责人失效处理
模板若配置了 `default_assigned_to`，在手动套用或自动生成时都必须重新校验当前有效性：
- 用户不存在 / 已停用 → 不再沿用该负责人，按未分配处理
- 团队模板下该用户已不是团队成员 → 不再沿用该负责人，按未分配处理

第一版不因默认负责人失效而让整个生成失败，避免周期任务因人员变动整体中断。

#### 模板启停与 schedule 启停关系
- `task_templates.is_active = false`：表示模板整体停用，既不能手动套用，也不能自动生成
- `template_schedules.enabled = false`：表示模板仍可手动套用，但不会自动生成
- 删除 schedule：表示该模板没有任何自动生成规则

这样模板层负责“这个模板还能不能被使用”，schedule 层负责“是否自动生成”。

### D4: 权限和边界规则

#### personal 模板
- 仅模板创建者可查看、编辑、删除、配置周期规则
- 其他用户不可见

#### team 模板
- 团队成员可查看并手动套用
- 团队负责人和管理员可创建、编辑、删除、配置周期规则
- 非团队成员不可见

#### system 模板
- 所有登录用户可查看并手动套用
- 仅管理员可创建、编辑、删除
- 第一版不允许配置周期规则

这个边界的核心是：系统模板是“公共标准动作”，不是“系统自动给所有人发任务”。自动生成只先落在个人和团队两个最明确的归属场景里。

### D5: 第一版页面改动方案

第一版前端只新增 3 个入口和 2 类配置界面，不改现有任务详情主流程。

#### 入口 1：我的任务页
在 `MyTasks.vue` 的头部区域新增“模板”入口，支持：
- 查看个人模板
- 新建/编辑个人模板
- 从模板快速创建任务
- 给个人模板配置周期规则

#### 入口 2：团队任务页
在 `TeamTasks.vue` 的头部区域新增“团队模板”入口，支持：
- 查看当前团队可用模板
- 团队负责人/管理员新建团队模板
- 从团队模板创建任务
- 给团队模板配置周期规则

#### 入口 3：后台管理页
新增与现有后台页面同级的“系统模板管理”页面，仅管理员可见，支持：
- 创建系统模板
- 编辑模板
- 启用/停用模板
- 手动套用说明

第一版不在后台支持系统模板的周期规则配置。
同时需要补齐两处集成：
- 在现有后台路由中新增系统模板管理路由
- 在管理员侧边导航中新增对应入口，并沿用当前 `requireAdmin` 的可见性与守卫模式

#### 两类界面
1. **模板列表页 / 抽屉 / 弹窗**
   - 显示模板名、默认标题、优先级、标签、范围、是否启用、是否已配置周期规则
   - 每条模板提供：套用创建、编辑、启停、删除、周期规则设置（仅个人/团队）
2. **模板编辑表单**
   - 字段尽量贴近现有任务创建表单：模板名、默认标题、描述、优先级、标签、默认负责人、默认截止偏移天数

#### 周期规则配置方式
周期规则单独做一个小弹窗，不与模板主表单混在一起。第一版只保留：
- 是否启用
- 频率：每天 / 每周 / 每月
- 小时
- 每周几 / 每月几号（按频率显示）

#### UX 细节
- “新建任务”按钮旁增加“从模板创建”或模板入口
- 空状态直接解释模板和周期任务用途
- 从模板创建任务时先弹出确认/编辑表单，允许用户修改后再创建
- 这个确认表单第一版允许修改的字段应与现有任务创建主表单尽量一致，至少包括：`title`、`description`、`priority`、`due_date`、`labels`，以及在有权限且上下文允许时的 `assigned_to`

### D6: 接口设计和前后端交互流程

建议围绕“模板”和“周期规则”两个核心对象设计接口。

#### 模板接口
- `GET /api/v1/task-templates?scope=personal`
- `GET /api/v1/task-templates?scope=team&team_id=12`
- `GET /api/v1/task-templates?scope=system`
- `POST /api/v1/task-templates`
- `PUT /api/v1/task-templates/{id}`
- `DELETE /api/v1/task-templates/{id}`
- `POST /api/v1/task-templates/{id}/instantiate`

其中 `instantiate` 单独作为动作接口，专门处理“用模板创建任务”，保持语义清晰。支持少量覆盖字段，但应尽量贴近现有任务创建体验，第一版建议支持：
- `title`
- `description`
- `priority`
- `due_date`
- `labels`
- `assigned_to`

后端统一流程：读取模板 → 校验权限与模板启用状态 → 组装默认任务数据 → 对默认负责人重新校验有效性 → 应用覆盖字段 → 按现有任务创建规则创建真正任务 → 返回新任务。

#### 周期规则接口
- `GET /api/v1/task-templates/{id}/schedule`
- `PUT /api/v1/task-templates/{id}/schedule`
- `DELETE /api/v1/task-templates/{id}/schedule`

`PUT` 既负责首次创建也负责后续更新，减少前端分支判断。也就是说，第一版明确采用 **upsert** 语义：
- 有现有 schedule 时更新
- 没有现有 schedule 时创建
- 不返回“重复创建 schedule”的冲突错误

请求字段建议包含：
- `enabled`
- `frequency`
- `hour`
- `weekday`
- `monthday`

#### 前端交互流程
1. **新建模板**：打开模板列表 → 新建模板 → 提交保存 → 刷新列表
2. **从模板创建任务**：点击套用 → 打开带默认值的任务表单 → 允许用户调整 → 提交实例化接口 → 刷新列表或跳转详情
3. **配置周期规则**：点击周期规则 → 拉取现有规则 → 编辑保存 → 列表状态更新

#### 后端自动生成流程
自动生成与手动套用最终都走同一套“模板实例化”为任务的内部逻辑，避免两套任务创建逻辑分叉。

#### 第一版错误处理
仅处理几类明确错误：
- 403：无权限
- 400：参数非法 / scope 与上下文不匹配 / 系统模板配置 schedule / 频率与 weekday 或 monthday 组合不合法 / 模板已停用
- 404：模板不存在

前端只需要把这些错误直接 toast 出来即可，不需要复杂错误页。

### D7: 测试方案和第一版实施顺序

#### 后端测试重点
分成三类：

1. **模板权限测试**
- personal：本人可管，其他人不可见
- team：团队负责人/管理员可管理，成员只读，非成员不可见
- system：管理员可管理，普通用户只读可套用
- 团队负责人变更后，后续团队模板管理权限应自动跟随新的 `team.created_by`

2. **模板实例化测试**
- 从个人/团队/系统模板成功创建任务
- 模板默认字段正确带入任务
- 覆盖字段优先生效
- 无权限时禁止实例化
- 模板不存在或失效时返回正确错误
- 默认负责人失效时自动回退为未分配，而不是整次创建失败

3. **周期生成测试**
- daily / weekly / monthly 命中逻辑正确
- disabled 不生成
- 同一时间窗口不重复生成
- system 模板不能配置 schedule
- 删除 schedule 后不再生成
- monthly 遇到 29/30/31 的短月时按“该月跳过”处理
- 服务停机恢复后不会补跑错过的周期

如果不适合直接测 APScheduler 本身，则将“是否命中 + 执行生成”的核心逻辑提炼为可测试函数单测。

#### 前端验证策略
第一版以前端手工验证和 build 校验为主：
- 个人 / 团队 / 系统模板入口可访问
- 能创建、编辑、删除模板
- 能从模板创建任务
- 个人/团队模板可配置周期规则
- 系统模板不显示周期规则入口
- 不破坏现有登录、任务列表、团队页、后台页

#### 推荐实施顺序
1. 后端模型和模板 CRUD
2. 模板实例化接口与内部复用逻辑
3. 前端模板入口与模板操作界面
4. 周期规则接口和后端自动生成逻辑
5. 前端周期规则弹窗

这样每一步完成后都能形成真实可用增量，不会出现底层做了很多但用户无法使用的情况。

## 不变的部分

- 现有任务详情、评论、附件主流程不改
- 现有任务状态流转规则不改
- 当前登录和刷新令牌机制不改
- Element Plus、Vue Router、Pinia、FastAPI、SQLAlchemy、APScheduler 的主技术栈不变
